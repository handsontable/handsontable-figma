# CONVENTIONS.md

## Source files

- **ESM everywhere.** Use `import`/`export`, include the `.js` extension in every import path (Node ESM requirement).
- **Double quotes** in source (`.prettierrc` pins `singleQuote: false`).
- **`printWidth: 120`**, 2-space indent, `trailingComma: "es5"`, semicolons required, `arrowParens: "always"`.
- Run `npm run format` before committing; there is no CI lint step, so Prettier is the only gate.
- **No side effects at module top level** except in `index.js` (which calls `main()`).

## Naming

- **Hyphen-case** throughout the *token domain* — that is what Figma exports and what `tokensKeys.js`, CSS generation, and raw tokens.json paths all use.
- **camelCase** in *generated JS output* only — applied by `toCamelCase` / `convertKeysToCamelCase` in `utils/jsGeneration.js`.
- Hyphen-before-digit → underscore (`colors-100` → `colors_100`). This is unusual — don't "fix" it; `handsontable` expects it.
- **CSS custom properties** are always prefixed `--ht-` (see `PREFIX` in `utils/constants.js`). The `tokens.` segment of a dotted reference is stripped from the CSS variable name (`tokens.background-color` → `--ht-background-color`), while `colors.`, `sizing.`, `density.` are kept (`colors.primary-100` → `--ht-colors-primary-100`).

## Token value conventions

- Every value that is a reference to another token looks like `{path.to.value}` in `tokens.json` and becomes a dotted string (`"tokens.background-color"`, `"colors.main-100"`) in `themeVariables`.
- Numeric values are unitless in `tokens.json`. `formatValue` adds the unit based on the key name:
  - Key contains `opacity` → `%`
  - Key contains `transition` → `s`
  - Otherwise → `px`
- Any key that should bypass unit formatting must be added to `EXCEPTION_KEYS` in `utils/constants.js`. Today that list contains only `"font-family"`.

## Adding a new token

1. Add the hyphen-case key to `tokensKeys.js` in the appropriate section (the file is grouped by feature area — keep groupings intact).
2. If the token needs a non-standard unit (not `px`/`%`/`s`), extend `formatValue` in `utils/themeProcessing.js` rather than hard-coding a unit in the token value.
3. If the token's *value* must pass through verbatim (e.g., a string like a font stack), add the key to `EXCEPTION_KEYS`.
4. Run `npm start` and diff `/output` — both the JS (`output/variables/tokens/<theme>.js`) and CSS (`output/css/theme/ht-theme-<theme>.css`) should pick up the new key automatically.

## Adding a new theme

1. Add the theme block to Figma and re-export `tokens.json`.
2. If the theme needs its own icon family, add an entry to `ICONS_SET` in `utils/constants.js` **and** update the icon-set selection branch in `utils/cssGeneration.js::generateThemeCss` (currently hardcoded: `horizon` → `horizon`, else → `main`).
3. Otherwise, the theme flows through automatically — `generateAllVariables` iterates `themes[THEME_KEY]`, so any new entry gets its own `output/variables/tokens/<name>.js` and `output/css/theme/ht-theme-<name>.css`.

## Adding or changing an icon

- Edit `icons/main.js` or `icons/horizon.js` — values are SVG `data:` URIs. The keys are referenced by string literal inside `utils/helpers/iconsMap.js` (`icon(icons, "arrowRight")`), so renaming an icon requires updating the map too.
- The icons-only CSS (`output/css/icons/ht-icons-<name>.css`) is generated by calling `iconsMap(icons)` with no theme prefix, so those stylesheets apply globally rather than scoped to a theme class.

## Output files

- Never hand-edit anything under `/output`. The `main()` function in `index.js` calls `rmSync(OUTPUT_PATH, { recursive: true })` before every regeneration.
- Every generated file begins with `AUTO_GENERATED_HEADER`; icon JS files additionally carry `/* eslint-disable max-len, quotes */` because they're consumed by a project with a stricter ESLint config.
